2.5 多分类学习

【目标】考虑N个类别C_1,C_2,…,C_n，多分类学习的思路是：将多分类任务拆为若干个二分类任务求解。先对问题进行拆分，然后为拆出的每个二分类任务训练一个分类器；在测试时，对这些分类器的预测结果进行集成以获得最终的多分类结果。
【拆分】一般有三种策略：“1对1”(OVO)、“一对其余”(OVR)、“多对多”(MVM)。给定数据集，这三种策略分别是这样操作的：

  1. **OVO **: 将N个类别两两配对，产生N(N-1)/2个二分类任务。测试时，新样本同时提交给所有的分类器，通过投票，把被预测得最多的类别作为最终分类结果。
  2. **OVR **: 每次将一个类的样例作为正例，其他所有类的样例作为反例，这个得到N个分类器。测试时若仅有一个预测为正类，那么它对应的类别标记作为最终分类结果。如图所示：

  1. **MVM **:每次将若干类作为正例，若干类作为反例。MVM需要特别设计，例如“纠错输出码”，即ECOC技术。EOCO技术分为两步：
    • 编码：对N个类别做M次划分，每次划分将部分类划为正类，部分划分为反类。可产生M个训练集。
    • 解码：M个分类器对样本进行预测，这些预测标记组成一个编码，返回预测编码和各类别各自编码距离最小的那个，作为最终结果。

  1. 信息技术的发展，对数据集的研究成为了各个领域的关键，数据成为了最重要和基础的资源，数据的发布、共享、分析成为了一个热门的话题

  2. 数据集中往往包含了诸多的隐私，隐私会随着数据的发布而被泄露。保护隐私，就是要使得任何用户访问数据库的过程中无法获取关于任何个体的确切信息。

  3. “K-匿名”方法被提出，通过分组的方式，在进行压缩、泛化等一系列处理后，将所有记录划分到若干等价类中，从而将单个记录隐藏在一整个类中。

  4. “K-匿名”方法存在两个问题：①攻击者所掌握的背景知识难以定义，总是因为出现新型的攻击方式而不断更新；②无法对隐私保护水平进行定量分析，从而也就无法用严格的数据来评估保护方法的能力。

  5. “差分隐私”方法被提出，相比“K-匿名”方法，存在两个特点：①假设攻击者能够掌握除了攻击目标之外的所有背景知识，默认以最恶劣的环境来处理问题；②对隐私保护进行了严格的定义，并提供了量化评估的方法。

2.2.4 差分隐私保护算法的组合性质

性质1 序列组合性
设有算法M1，M2，……，Mn，其隐私保护预算分别为ε1，ε2，……，εn，那么对于同一数据集D，由这些算法构成的组合算法M(M1(D),M2(D),……,Mn(D))，提供（Σεi）-差分隐私保护。
这种组合算法，预算为全体预算的总和。
性质2 并行组合性
设有算法M1，M2，……，Mn，其隐私保护预算分别为ε1，ε2，……，εn，那么对于不相交的数据集D1，D2……Dn，由这些算法构成的组合算法M(M1(D1),M2(D2),……,Mn(Dn))，提供（max ε）-差分隐私保护。
如果算法序列中，所有算法处理的数据集彼此不相交，那么预算为所有预算中最大值，也即隐私保护水平取决于算法序列中保护水平最差者。

2.1 评估方法

2.1.1 留出法

  • 基本概念：直接将数据集D分为两个互斥的集合S和T，在集合S上进行训练模型，在T上进行测试，结果的误差作为对“泛化误差”的估算。
  • 注意事项：保持样本的类别比例相似，采样时应该采取“分层采样”的方式。此外，单次使用留出法得出的结论往往不准确，需要进行若干次随机划分、重复实验评估后取均值来作为评估结果。
  • 包含问题：如果训练集过大，则测试集相对就小，评估结果可能不太准确、不太稳定；而如果减小训练集，扩大测试集，评估模型与训练结果差别更大，参考性就弱了。测试集小时，方差大；训练集小时，偏差大。

2.1.2 交叉验证法

  • 基本概念：将一个数据集D分为k组，训练时取其中k-1组，剩下一组作为测试集，如此可以进行k次训练，最终结果就是这k次训练结果的均值。交叉验证法因此可称为“k折交叉验证”。此外，受限于划分方法，训练时往往要随机使用多次划分重复p次，就成了“p次k折交叉验证”。
  • 特殊情况：当 k=|D| 时，即每次只拿出一个数据来测试，其余数据来进行训练，交叉验证就变成了“留一法”。这样的训练样本与原数据集非常相似，评估结果往往较为准确，但开销极大。

2.1.3 自助法

  • 基本概念：从数据集D中，随机选出一个样本，将其复制到数据集D’ 中，D集合不变。这样重复m次。当m足够大时，取极限 ，这个结果的意思是：m次采样都没有采中的概率。作数学期望，可知，当m足够大时，样本中仍旧有差不多36.8%的数据没有录入。因此可以用D’ 作训练集，D\D’ 作测试集。
  • 解决问题：注意到|D|=m，也即，实际数据集和期望数据集的大小都是m，这样可以解决数据集规模不同引发的估计偏差。在数据集较少时很有用。
  • 存在问题：自助法改变了数据的分布，这就又引入了分布不同产生的估计误差。

2.1.4 调参与自助模型

  • 基本概念：在进行模型评估与选择时，除了算法之外，参数的设定往往也会改变模型的结果。参数设定本质上与算法选取没什么区别，关键在于参数是在实数集上取的，因而无法每个结果都测一遍，只能以“范围+步长”的方式确定候选的参数，再从候选参数中选一个最合适的。
  • 注意事项：当训练集和验证集通过后，我们得到了我们的模型和参数设置，此时应该用原始的数据集D再训练一次，这样才能最终提交给用户。

答：先看看F1值的定义，

，其中 即查准率（Precision ），【预测为正例且真实为正例的数量】/【预测为正例的数量】，说白了关心预测为正样本时的准确率； 即查全率（又称召回率Recall），【预测为正例且真实为正例的数量】/【真实为正例的数量】。F1值计算中对查准率和查全率都同等重视。
再看看BEP值
首先目前很多分类算法输出的都是0-1之间的一个概率值，比如逻辑回归、xgboost等，分类时的做法是预定一个阈值（典型为0.5），若对样本的输出大于此阈值则归为1类（即正例），那么根据样本的输出值从大到小排序（下文简称为“样本的排序”），排在最前面的即可理解为最有可能为正例的样本，而排在最后的是最不可能为正例的样本。从前往后，逐个将样本预测为正例（即把当前样本的输出值定于为阈值，小于阈值的都为反例），每次计算当前的查准率和查全率，即可得到查全率为横坐标查准率为纵坐标上的一个点，在将所有点按顺利连接后即可得到“P-R曲线”，而BEP（即Break-Event Point，平衡点）是在查全率=查准率时的取值。

讨论：
从定义上看，F1值是在阈值固定时，将所有样本分类完成后，综合查全率和查准率得出的值；而BEP值则是寻求一个阈值使得查全率和查准率相同的情况下得到的（此时BEP = 查全率 = 查准率）。
也就是说BEP值和“样本的排序”紧密相关的，而和样本的预测值大小无关，同样的排序，即使将所有预测值同时乘以0.5，其BEP值也是相同的；但是对于F1值，所有样本都将预测为负例（假定阈值为0.5时），此时F1值为0。
回到题目本身，“若学习器A的F1值比学习器B高，则A的BEP值比B高”，那么若能找到两个学习器BEP值相同，而F1值不同，则题目命题就不成立了。那从上面的讨论中已经有了答案了，想象一下学习器A对样本输出值均为学习器B的两倍，两者BEP值是相同的，A的输出在（0,1）之间，而B的输出在（0,0.5）之间，此时B的 F1 值为0，A的 F1 值是在0-1之间。所以原命题不成立。
ps.个人从直觉上BEP值和F1值是没有明确关系的，在讨论过程中拿“输出值乘以0.5”为例，事实上，想象一下，一串固定排序的点(模型的输出概率值)，只在0-1之间同时前进或者后退（每个点前进步长可以不一样，但是排序不变），其BEP值也不会发生变化，而F1值是不断变化的。

2.1 引言部分

  1. 机器学习研究的内容，是关于“学习算法”的，即从“数据”中产生“模型”的算法。
  2. 一些专用术语：数据、记录、数据集、示例/样本、属性/特征、属性值、属性空间/样本空间/输入空间、特征向量（示例）、维数、训练数据、训练样本、训练集、假设、真实、学习器、标记、样例、标记空间。
  3. 如果预测的是离散值，则学习任务称为“分类”。如果是连续值，则称为“回归”。预测任务是希望通过对训练集进行学习，从而建立一个从输入空间X到输出空间Y的映射。模型建立起来后，使用其进行预测的行为，称为“测试”。
  4. 监督学习：分类和回归。无监督学习：聚类。
  5. 学得模型适用于新样本的能力，就是“泛化”能力。训练的目标就是使得模型的泛化能力更强。

2.2 假设空间

  1. 归纳：从特殊到一般，也就是“泛化”的步骤。归纳学习也可以分为“狭义”和“广义”，后者指从样例中学习，前者要求从训练数据中学得概念，前者目前的应用较少，大多是“黑箱”型的模型。
  2. 演绎：从一般到特殊，也就是“特化”的步骤。
  3. 概念学习中，最基本的是“布尔概念”的学习，也就是0/1值的目标概念的学习。
  4. 假设空间与版本空间，前者是所有维度中所有可能值组成的空间，后者是训练集的可能泛化集合。简单理解，版本空间就是软件测试中的“等价类划分”。

2.3 归纳偏好

  1. 训练集泛化，产生版本空间，而版本空间的存在，必然导致模型的“选择策略”问题。“归纳偏好”就诞生了。
  2. 如果没有“偏好”，则模型训练将随机抽取训练集的等效假设，这样对于某些特定的预测，结果可能完全相反，学习便没有了意义。
  3. 奥卡姆剃刀：若有多个假设与观察一致，则选最简单的那个。算法的归纳偏好与问题本身是否匹配，这个因素很多时候直接决定了算法的性能。
  4. 没有免费的午餐（NFL定理）：如果所有“问题”出现的机会相同、或所有问题同等重要，则在期望性上，学习算法的期望性能与随机胡猜都差不多。
  5. 要谈论算法的相对优劣，必须要针对具体的学习问题。

（3）一共有18种瓜，而每个瓜是否在析合范式中，只存在两种可能：是或者不是。因此，所有析合范式加起来的假设数量，不会超过$2^{18}$个（这里的意思是这样的，比如在范式A中，瓜1是好瓜，而在范式B中，瓜1不是好瓜。那么这两个范式就涵盖了瓜1的两种可能情形。以此类推，对于所有的范式，涵盖的可能性最多不超过$2^{18}$种。我们可以据此推断出析合范式包含的合取式的上限数量）。
（4）选取 i 个合取式，一共有$C_{48}^{i}$种不同的选取方式，每种选取方式都将决定18种瓜中哪些是好瓜，哪些是坏瓜。换而言之，每种选取方式决定了一个18维的向量的值（这个值仅为0或者1，用来决定这个瓜是不是好的）。
（5）因此，判断 i=k 时，需要遍历$C_{48}^{i}$种可能的情况，并将结果重复（18维向量相等）的筛掉。
（6）最终，对于i$\le$k的情况，依次做5的操作，最后再将结果叠加即可。

hexo g 指令无法创建index.html文件

发表于 2022-02-19 更新于 2022-02-20 分类于 技术
  1. 背景描述

    笔者新建了项目 “Solituder1969”, 使用 “hexo” 工具搭建博客。在将主题更换为 “next” 、添加了博主图片、首页新增 “tags” 和 “categorites” 分类后，本地4000端口 debug 没有任何问题。之后应该 清理缓存，重新生成，部署到 github 。

  2. 问题描述

    在第二步中，使用 “hexo generate” 指令，生成的 “public” 文件夹（也即要部署的静态文件夹）里却没有主页 “index.html”，对于 “tags”和”categories” 也没有生成对应的 “.html” 文件，只有初始的 “.md”文件。其结果就是本地再次调用 “hexo s” 指令运行时，只有一个 **_”Cannot get /“_**。

  3. 问题归因

    3.1 hexo generate 的原理
    浏览项目目录，我们发现 hexo 项目的结构是这样的：

    -node_modules
    -public
    -scaffolds
    -source
    -themes

    可以发现， source 文件夹与 themes 文件夹处于同级目录， public 为生成的静态包， node_modules 为 npm 引入的模块， scaffolds 则提供了”draft”、”page”和”post”三种格式。

    根据 “模板引擎”的理论，我们可以简单地认为， source 中存放了我们博客中的数据（标签、文章等）， themes 则为整个博客使用的模板。

    hexo 先遍历 themes 下主题（例如 “next”）的 source 文件，建立索引，随后根据主目录下 source 里存放的文章类型，以 layout 里对应的布局将文章渲染成html文件。

    至于hexo是如何遍历、如何进行数据填充的，这就涉及到插件 hexo-generator-index 了。它会首先调取 _config.yml 里的参数，然后指定对象需要填充进哪个路径中，最后再通过page变量调取。

    通过上述的分析，我们知道， “hexo generate” 指令生成主页，一共需要以下四个方面支持：

    a. 数据源，原本是什么样子的
    b. 模板引擎，要生成什么样子的
    c. 配置文件，a和b中元素是如何对应的，模板的具体细节
    d. 插件，a和b的对应关系是如何填充的

    3.2 问题诊断

    数据源 使用 _“hexo new hello-world”_ 指令生成了一篇文章，确保主页不空
    模板引擎 套用的是 _“next”_ 模板，这是当前比较受欢迎的博客模板
    配置文件 更改了相关的参数属性，但仅涉及诸如头像、进度条、搜索等功能性相关
    插件 齐全，没有报错
  4. 解决方案
    这里就是比较离奇的地方了。笔者在尝试了4个多小时之后，仍旧没有解决这个问题。之后笔者重新利用 hexo init 指令，初始化了一个 text 项目，并将原项目中的 config 文件直接覆盖了过来，安装插件之后，_test_ 项目经过 clean 后，居然可以 generate 出 index.html 了！

    原项目与测试项目除了名字之外，其余都是一样的。但原项目就无法生成主页，而测试项目就可以无问题产出结果。不得不说，挺奇妙的……
  5. 问题总结
    A. 发现了 hexo 构建项目时的一个问题：无法生成默认页；
    B. 尝试着分析了一下问题的源头，主要来自数据源、模板引擎、配置文件和插件四个方面；
    C. 尝试了4个小时，反复排查了过去，没有能够解决问题；
    D. 重新构建了一个测试项目，并把原工程的相关文件覆盖了一遍，可以顺利运行产出预期结果；
    E. 已成功部署，但问题仍待解决……
